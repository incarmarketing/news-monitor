# 관심키워드 및 모니터링 자동화

이 문서는 **처음 보는 사람도 바로 이해할 수 있게 만든 매뉴얼**입니다.

이 프로젝트는 내가 관심 있는 키워드를 정해두면, 관련 자료를 자동으로 찾고, 관련 없는 내용을 걸러내고, AI가 요약과 분류를 한 뒤, 대시보드와 보고서와 알림으로 보여주는 자동 모니터링 도구입니다.

현재는 인카금융서비스 마케팅부의 언론/PR 모니터링 업무를 기준으로 만들어졌지만, 구조는 단순합니다. 뉴스 대신 보도자료, 공시, 블로그, 경쟁사 홈페이지, 채용공고, 고객 리뷰 등을 모으면 다른 업무 모니터링 도구로도 확장할 수 있습니다.

```text
관심 키워드 입력
→ 자동 검색
→ 관련 없는 자료 제거
→ AI 요약/분류
→ Supabase 저장
→ 대시보드, 보고서, 카카오 알림으로 확인
```

## 1. 이걸 왜 만들었나요?

언론홍보 업무에서 가장 귀찮은 일은 매일 같은 키워드를 직접 검색하고, 기사 하나하나를 열어보고, 우리 회사와 관련 있는지 판단하고, 부정 이슈가 있는지 다시 확인하는 일입니다.

특히 이런 일이 반복됩니다.

- 네이버, 구글에서 같은 키워드를 계속 검색한다.
- 같은 기사가 여러 매체에 반복 노출되는지 확인한다.
- 오래된 기사가 다시 검색된 것인지 구분한다.
- 우리 회사 직접 언급인지, 단순 업계 기사인지 판단한다.
- 긍정, 중립, 주의, 부정 기사로 나눈다.
- 부정 기사나 규제 이슈가 있으면 빨리 알아야 한다.
- 아침, 점심, 퇴근 전에 보고서로 정리해야 한다.
- 주간, 월간 흐름도 따로 보고 싶다.

이 프로젝트는 위 일을 자동화해서 **검색 노동을 줄이고, 놓치면 안 되는 신호를 빨리 확인하고, 누적 데이터로 보고서까지 만들기 위해** 만들었습니다.

## 2. 아주 쉽게 이해하기

이 시스템은 네 가지 일을 합니다.

1. **찾기**

   내가 등록한 키워드로 뉴스와 보도자료를 찾습니다.

2. **걸러내기**

   중복 기사, 오래된 기사, 업무와 관계없는 기사를 최대한 제외합니다.

3. **분석하기**

   AI가 기사를 읽고 당사, GA, 보험사, 정책/규제 같은 기준으로 나누고, 긍정/중립/주의/부정 논조를 판단합니다.

4. **보여주기**

   대시보드, 보고서, 스크랩, 리스크 대응센터, 카카오 알림으로 확인할 수 있게 정리합니다.

## 4. 화면별 사용법

### 통합 대시보드

가장 먼저 보는 화면입니다.

여기서는 현재 모니터링 상태를 한눈에 봅니다.

- 수집 기사 수
- 분석 기사 수
- 당사 언급 기사
- 당사 부정 기사
- 주의 기사
- 주요 이슈
- 분류별 흐름
- 알림톡 발송 이력
- 부정기사 감시 상태

`갱신` 버튼을 누르면 최신 데이터를 다시 불러와 화면에 반영합니다.

### 실시간 모니터링

수집된 기사 목록을 자세히 보는 화면입니다.

여기서는 다음을 할 수 있습니다.

- 날짜 기준으로 기사 보기
- 긍정, 중립, 주의, 부정, 제외 기준으로 필터링
- 당사, GA, 보험사, 정책/규제 등 분류별 보기
- 기사 원문 열기
- 중요한 기사 스크랩
- AI 요약 확인
- 관련 기사 묶음 확인

기사 목록은 최신 흐름을 보기 위한 화면이므로, 너무 오래된 기사보다 현재 구간의 이슈를 확인하는 데 집중합니다.

### 금융당국 보도자료

금융감독원, 금융위원회 등 금융당국의 보도자료를 모니터링하는 화면입니다.

여기서는 다음을 할 수 있습니다.

- 최신 보도자료 갱신
- 보험, GA, 설계사, 수수료, 내부통제 관련 보도자료 확인
- 선택한 보도자료만 모아서 AI 분석 요청
- 보도자료와 관련된 뉴스 보기

뉴스보다 보도자료는 정책 방향을 먼저 보여주는 경우가 많아서, 리스크와 제도 변화 확인에 중요합니다.

### 미디어 분석 리포트

기간을 정해서 언론 흐름을 분석하는 화면입니다.

여기서는 다음을 볼 수 있습니다.

- 선택 기간의 전체 기사량
- 당사 기사
- 부정/주의 논조
- 정책/규제 이슈
- 언론사 활동
- 키워드별 기사량
- 핵심 이슈
- 분석 코멘트

일간, 주간, 월간 기준으로 볼 수 있고, 필요하면 보고서 형태로 인쇄하거나 PDF로 저장합니다.

### 주요 기사 스크랩

실시간 모니터링에서 스크랩한 기사를 모아보는 화면입니다.

여기서는 원하는 방향으로 AI 분석을 요청할 수 있습니다.

예를 들어 이렇게 요청할 수 있습니다.

```text
임원 보고용으로 핵심 판단, 리스크, 후속 확인 포인트를 간결하게 정리해줘.
```

```text
이 기사들이 당사 평판에 어떤 영향을 줄 수 있는지 정리해줘.
```

```text
영업 현장에서 문의가 들어왔을 때 설명할 수 있게 정리해줘.
```

### 리스크 대응센터

부정 또는 주의 기사에 대응하기 위한 화면입니다.

여기서는 다음을 할 수 있습니다.

- 부정/주의 기사 전체 확인
- 기사 URL 붙여넣기 또는 드래그
- 이슈 내용 입력
- AI 대응 초안 생성
- 사내 보고용, 언론 대응용 문안 정리

현재 AI 초안은 Gemini Pro 기반으로 생성합니다.

### 언론사 관리

언론사별 보도량과 관계 정보를 관리하는 화면입니다.

여기서는 다음을 할 수 있습니다.

- 언론사별 보도량 확인
- 당사 언급 수 확인
- 부정/긍정 흐름 확인
- 담당자, 등급, 메모 관리
- 주소가 잘못 잡힌 언론사명 보정

네이버나 구글은 언론사가 아니기 때문에, 실제 원문 언론사명으로 보정하는 기능이 중요합니다.

### 기자 관리

기자 정보를 관리하는 화면입니다.

기자 목록을 항상 길게 펼쳐두지 않고, 필요한 기자만 `관리` 버튼으로 열어 수정하는 방식입니다.

### 광고비/협찬 관리

광고비와 협찬 집행 내역을 관리하는 화면입니다.

여기서는 다음을 볼 수 있습니다.

- 월별 집행액
- 연도별 집행 흐름
- 언론사별 집행 현황
- 구분별 집행 현황
- 보고서 인쇄용 화면

언론사 관계 관리와 예산 집행 현황을 함께 보기 위한 기능입니다.

### 환경 설정

모니터링 키워드를 관리하는 화면입니다.

여기서 키워드를 추가하거나 삭제하면, Supabase에 저장된 키워드와 동기화됩니다.

즉, 화면에서 키워드를 바꾸면 이후 수집과 분석 조건에도 반영되는 구조입니다.

## 5. 키워드 넣는 법

키워드는 너무 짧으면 엉뚱한 기사를 많이 가져옵니다.

예를 들어 `메가`만 넣으면 GA 회사 메가가 아니라 메가톤급, 메가 이벤트, 메가박스 같은 기사까지 들어올 수 있습니다.

좋은 키워드는 업종 맥락이 같이 들어간 키워드입니다.

좋은 예시는 다음과 같습니다.

- 인카금융서비스
- 인카금융서비스 브랜드평판
- 보험설계사
- GA 보험
- 보험대리점
- 보험대리점 브랜드평판
- 설계사 근로자성
- 보험 수수료
- 1200%룰
- 금융감독원 보험
- 내부통제 보험
- GA 리크루팅

조심해야 하는 키워드는 다음과 같습니다.

- 메가
- 프로
- 수수료
- 채용
- 플랫폼

이런 단어를 꼭 써야 한다면 업종 단어를 붙이는 것이 좋습니다.

```text
메가 GA
보험 수수료
설계사 채용
보험 플랫폼
```

## 6. 자동화 시간표

자동 실행은 GitHub Actions와 cron-job.org를 이용합니다.

PC가 꺼져 있어도 클라우드에서 실행되도록 구성했습니다.

### 일일 언론동향

- 08:00 발송: 전일 18:00부터 당일 08:00까지
- 13:00 발송: 당일 08:00부터 13:00까지
- 18:00 발송: 당일 13:00부터 18:00까지

### 부정기사 감시

- 5분마다 실행
- 부정 또는 주의 신호가 있는지 확인
- 필요 시 카카오 알림 발송

### 주간 보고서

- 매주 월요일 07:00 발송
- 전주 월요일부터 일요일까지의 흐름을 정리

### 월간 보고서

- 매월 1일 07:00 발송
- 전월 전체 흐름을 정리

### 금융당국 보도자료

- 매일 07:30 기준으로 갱신
- 금융감독원, 금융위원회 보도자료를 확인

## 7. 데이터는 어디에 저장되나요?

이 프로젝트는 여러 도구가 역할을 나눠서 움직입니다.

### Supabase

데이터를 저장하는 곳입니다.

쉽게 말하면 이 프로젝트의 데이터베이스입니다.

저장되는 내용은 다음과 같습니다.

- 수집 기사
- AI 분석 결과
- 모니터링 키워드
- 언론사 정보
- 기자 정보
- 광고비/협찬 내역
- 알림톡 발송 이력
- 스크랩 정보

### GitHub Pages

대시보드와 보고서를 보여주는 곳입니다.

브라우저에서 접속하는 화면은 GitHub Pages에 올라갑니다.

### GitHub Actions

자동 실행을 담당합니다.

정해진 시간마다 뉴스 수집, 분석, 보고서 생성, 알림 발송을 실행합니다.

### Kakao API

카카오톡으로 알림을 보내는 데 사용합니다.

### Gemini API

기사 요약, 논조 판단, 리스크 대응 초안, 스크랩 분석에 사용합니다.

## 8. 수동으로 다시 돌리고 싶을 때

### 대시보드에서 바로 확인

통합 대시보드나 실시간 모니터링 화면에서 `갱신` 버튼을 누릅니다.

### 금융당국 보도자료 다시 가져오기

금융당국 보도자료 화면에서 `최신 보도자료 갱신` 버튼을 누릅니다.

### GitHub Actions에서 직접 실행

GitHub 저장소에 들어가서 `Actions` 메뉴를 누른 뒤 원하는 작업을 직접 실행할 수 있습니다.

처음에는 어렵게 느껴질 수 있지만, 구조는 단순합니다.

```text
GitHub 저장소
→ Actions
→ 실행할 workflow 선택
→ Run workflow
```

## 9. 문제가 생겼을 때

### 화면이 안 바뀌어요

브라우저 캐시 때문에 예전 화면이 보일 수 있습니다.

먼저 새로고침을 해보세요.

그래도 안 되면 GitHub Actions 배포가 끝났는지 확인합니다.

### 카카오 알림이 안 와요

다음을 확인합니다.

- GitHub Actions가 실행됐는지
- Kakao refresh token이 만료되지 않았는지
- Supabase에 발송 이력이 남았는지
- 알림을 보낼 조건이 실제로 있었는지

### 이상한 기사가 들어와요

대부분 키워드가 너무 넓거나 문맥 필터가 부족한 경우입니다.

예를 들어 `수수료`만 넣으면 보험 수수료가 아니라 배달 수수료, 플랫폼 수수료 기사도 들어올 수 있습니다.

이럴 때는 키워드를 더 구체적으로 바꿉니다.

```text
수수료 → 보험 수수료
메가 → 메가 GA
설계사 → 보험설계사
```

### 언론사 이름이 주소로 보여요

언론사 관리 화면에서 해당 언론사를 열고 주소 보정을 등록합니다.

예를 들어 `insnews.co.kr`이 보험매일이면, 해당 주소를 보험매일로 저장합니다.

이후 같은 주소로 들어오는 기사는 보험매일로 표시되도록 관리합니다.

### 요약이 이상해요

원문이 너무 짧거나, RSS에 본문이 거의 없거나, 기사 첫 문장이 사진 설명인 경우 요약 품질이 떨어질 수 있습니다.

이 경우 원문 본문을 더 잘 가져오도록 수집 로직을 개선하거나, 스크랩 분석과 리스크 대응센터에서 별도 분석을 요청합니다.

## 10. 보안 주의

API 키와 토큰은 절대 README, GitHub 이슈, 카카오톡, 채팅창에 그대로 올리면 안 됩니다.

민감한 값은 아래 위치에만 저장합니다.

- GitHub Secrets
- Supabase Secrets
- 로컬 `.env` 파일

특히 아래 값은 외부에 공유하면 안 됩니다.

- Gemini API Key
- Kakao REST API Key
- Kakao Refresh Token
- Supabase Service Role Key
- GitHub Token
- cron-job.org API Key

## 11. 다른 업무로 확장하는 법

이 프로젝트는 언론 모니터링으로 시작했지만, 본질은 관심 키워드 자동 모니터링입니다.

수집 대상을 바꾸면 다른 업무에도 쓸 수 있습니다.

예를 들어 다음처럼 확장할 수 있습니다.

- 경쟁사 홈페이지 모니터링
- 금융당국 보도자료 모니터링
- 채용공고 모니터링
- 공시 모니터링
- 고객 리뷰 모니터링
- 블로그나 커뮤니티 반응 모니터링
- 특정 상품명 언급량 추적
- 특정 법안이나 규제 키워드 추적

바꿔야 할 것은 네 가지입니다.

1. 어디에서 자료를 가져올지
2. 어떤 키워드로 찾을지
3. 어떤 기준으로 분류할지
4. 어떤 보고서 양식으로 보여줄지

## 12. 그대로 따라 하는 상세 실행 프로세스

여기부터는 실제 운영자가 그대로 따라 할 수 있는 상세 매뉴얼입니다.

앞부분이 “이 도구가 무엇인지”를 이해하기 위한 설명이라면, 이 부분은 “그래서 오늘 업무에서 어떻게 쓰는지”를 순서대로 정리한 내용입니다.

### 12-1. 매일 아침 확인 절차

아침에는 전일 18:00 이후부터 당일 08:00까지 나온 기사와 보도자료를 확인합니다.

1. 대시보드에 접속합니다.

   https://incarmarketing.github.io/news-monitor/dashboard.html

2. 통합 대시보드 상단의 주요 숫자를 확인합니다.

   - 수집 기사
   - 분석 기사
   - 당사 언급
   - 당사 부정
   - 주의 기사

3. `LIVE MEDIA BRIEFING` 문장을 먼저 읽습니다.

   이 문장은 현재 상황을 한 줄로 요약한 판단 문장입니다.

4. `주요 이슈` 영역을 확인합니다.

   여기에는 단순 최신 기사보다 우선순위가 높은 기사가 올라와야 합니다.

   우선순위는 보통 다음 순서입니다.

   1. 당사 직접 부정 기사
   2. 당사 직접 언급 기사
   3. 금융감독원, 금융위원회, 제도, 규제 관련 기사
   4. GA 또는 보험대리점 업계 변화
   5. 주요 보험사, 경쟁사 동향
   6. 브랜드평판, 실적, 제휴, 채널 변화

5. `알림톡 발송 이력`을 확인합니다.

   가장 최근 발송 내역 3개가 먼저 보여야 합니다.

   하루 일일보고가 08:00, 13:00, 18:00 세 번이므로 기본 화면에서도 최소 3개는 바로 보여야 합니다.

6. 이상이 없으면 실시간 모니터링으로 이동해 기사 목록을 자세히 확인합니다.

### 12-2. 점심과 퇴근 전 확인 절차

점심과 퇴근 전에는 새로 쌓인 기사 흐름을 봅니다.

점심 보고 기준은 보통 당일 08:00부터 13:00까지입니다.

퇴근 전 보고 기준은 보통 당일 13:00부터 18:00까지입니다.

확인 순서는 아침과 같습니다.

1. 통합 대시보드 접속
2. 주요 숫자 확인
3. 주요 이슈 확인
4. 실시간 모니터링에서 기사 원문 확인
5. 필요 기사 스크랩
6. 리스크가 있으면 리스크 대응센터에서 초안 생성
7. 알림톡 발송 이력 확인

중요한 점은 일일보고가 “하루 전체 기사”를 무조건 다 보여주는 것이 아니라, 해당 시간 구간의 핵심 변화와 리스크를 보여줘야 한다는 점입니다.

### 12-3. 관심 키워드 추가 방법

새 키워드를 추가할 때는 환경 설정 화면을 사용합니다.

1. 왼쪽 메뉴에서 `환경 설정`을 클릭합니다.
2. `모니터링 키워드` 영역으로 이동합니다.
3. 키워드 구분을 선택합니다.

   현재 기준은 보통 다음처럼 나눕니다.

   - 당사
   - GA
   - 보험사
   - 정책/규제
   - 제외 후보

4. 키워드를 입력합니다.
5. `추가`를 누릅니다.
6. `Supabase 동기화` 또는 저장 상태를 확인합니다.
7. 이후 수집/분석에 새 키워드가 반영되는지 실시간 모니터링에서 확인합니다.

키워드는 너무 짧게 넣지 않는 것이 중요합니다.

예를 들어 `메가`는 너무 넓습니다.

이 경우 아래처럼 바꾸는 것이 좋습니다.

```text
메가 → 메가 GA
메가 → 메가금융서비스
수수료 → 보험 수수료
설계사 → 보험설계사
정책 → 보험 정책
```

### 12-4. 키워드를 넣을 때 판단 기준

키워드는 “검색어”이면서 동시에 “분석 기준”입니다.

그래서 아무 단어나 넣으면 안 됩니다.

좋은 키워드는 아래 조건을 만족합니다.

- 우리 회사, 업계, 경쟁사, 제도와 관련이 있다.
- 너무 일반적인 단어가 아니다.
- 검색했을 때 엉뚱한 분야가 너무 많이 섞이지 않는다.
- 기사 제목이나 본문에서 실제로 자주 쓰이는 표현이다.

예를 들어 좋은 키워드는 다음과 같습니다.

```text
인카금융서비스
인카금융
보험설계사
GA 보험
보험대리점
보험대리점 브랜드평판
설계사 근로자성
정착지원금
1200%룰
보험 수수료
금융감독원 보험
내부통제 보험
불완전판매
```

반대로 단독으로 쓰기 위험한 키워드는 다음과 같습니다.

```text
메가
수수료
채용
플랫폼
프로
관리
```

이런 단어는 반드시 업종 단어와 묶어서 씁니다.

### 12-5. 수동으로 최신 기사 갱신하는 방법

자동화가 5분마다 돌더라도, 바로 확인하고 싶을 때는 수동 갱신을 사용할 수 있습니다.

1. 통합 대시보드로 이동합니다.
2. `주요 이슈` 영역의 `갱신` 버튼을 누릅니다.
3. 버튼을 누르면 최신 키워드 기준 수집과 분석 요청이 들어갑니다.
4. GitHub Actions 또는 Supabase Function이 실행됩니다.
5. 잠시 후 화면이 다시 최신 데이터로 바뀝니다.

수동 갱신은 “지금 바로 다시 찾아봐”에 가깝습니다.

다만 수집, 분석, 저장, 재조회까지 시간이 조금 걸릴 수 있습니다.

### 12-6. 실시간 모니터링 보는 방법

실시간 모니터링은 기사 목록을 직접 보는 화면입니다.

사용 순서는 다음과 같습니다.

1. 왼쪽 메뉴에서 `실시간 모니터링`을 클릭합니다.
2. 기준일을 확인합니다.
3. 논조 필터를 선택합니다.

   보통 확인 순서는 아래가 좋습니다.

   1. 부정
   2. 주의
   3. 당사
   4. 정책/규제
   5. 전체

4. 필요한 경우 분류 필터를 선택합니다.

   - 당사
   - GA
   - 보험사
   - 정책/규제
   - 기타

5. 기사 제목, 언론사, 요약을 확인합니다.
6. 원문 확인이 필요하면 `기사 열기`를 누릅니다.
7. 계속 봐야 할 기사는 `스크랩`을 누릅니다.

실시간 모니터링에서 중요한 것은 모든 기사를 다 읽는 것이 아닙니다.

당사, 부정, 주의, 정책/규제, 업계 구조 변화 기사부터 먼저 보는 것이 핵심입니다.

### 12-7. 기사 논조 판단 기준

AI가 기사를 분류하더라도, 운영자는 기준을 이해하고 있어야 합니다.

### 긍정

당사에 우호적이거나 활용 가능한 기사입니다.

예시는 다음과 같습니다.

- 당사 실적 개선
- 브랜드평판 1위
- 긍정적 제휴
- 신사업 성과
- 수상, 인증, 평가 개선

단, 당사가 언급되지 않은 경쟁사 성과 기사는 긍정이 아니라 중립 또는 업계동향입니다.

### 중립

사실 전달 중심의 기사입니다.

예시는 다음과 같습니다.

- 보험사 일반 동향
- GA 시장 변화
- 단순 제휴 기사
- 당사와 직접 관련 없는 업계 기사

### 주의

직접 부정은 아니지만 별도 관찰이 필요한 기사입니다.

예시는 다음과 같습니다.

- 당사 주가 하락 또는 시황성 언급
- 감독, 검사, 제도 변화
- 정착지원금, 수수료, 내부통제 이슈
- GA 시장 위축
- 당사와 같은 업종의 부정 이슈

### 부정

당사에 직접적인 평판 리스크가 있거나, 소비자 피해, 제재, 법적 분쟁, 불완전판매 등 명확한 리스크가 있는 기사입니다.

다만 당사가 직접 언급되지 않은 일반 보험사기 기사나 타사 부정 기사는 무조건 당사 부정으로 보지 않습니다.

### 제외

업무와 관계없는 기사입니다.

예시는 다음과 같습니다.

- 보험과 무관한 수수료 기사
- 정치, 지역, 생활 기사 중 키워드만 우연히 겹친 기사
- 오래된 기사의 단순 재노출
- 사진 설명이나 광고성 문구만 있는 자료

### 12-8. 스크랩 기능 사용 방법

스크랩은 “나중에 다시 봐야 하는 기사”를 모으는 기능입니다.

1. 실시간 모니터링에서 중요한 기사를 찾습니다.
2. `스크랩` 버튼을 누릅니다.
3. 주요 기사 스크랩 화면으로 이동합니다.
4. 스크랩한 기사들이 모여 있는지 확인합니다.
5. 분석 요청창에 원하는 분석 방향을 입력합니다.

예시는 다음과 같습니다.

```text
임원 보고용으로 핵심 판단, 리스크, 후속 확인 포인트를 간결하게 정리해줘.
```

```text
이 기사들이 당사 평판에 어떤 영향을 줄 수 있는지 분석해줘.
```

```text
영업 현장에서 문의가 들어올 경우 답변 방향을 정리해줘.
```

AI 분석은 스크랩한 기사만 대상으로 하기 때문에, 보고용 정리에 유용합니다.

### 12-9. 금융당국 보도자료 확인 방법

금융당국 보도자료는 뉴스보다 먼저 제도 변화를 보여줄 수 있습니다.

확인 순서는 다음과 같습니다.

1. 왼쪽 메뉴에서 `금융당국 보도자료`를 클릭합니다.
2. `최신 보도자료 갱신`을 누릅니다.
3. 금융감독원, 금융위원회 관련 보도자료를 확인합니다.
4. 보험, GA, 설계사, 수수료, 내부통제 관련 자료를 우선 봅니다.
5. 필요한 자료를 체크합니다.
6. 분석 요청창에 원하는 분석 방향을 입력합니다.
7. `선택 자료 분석`을 누릅니다.

추천 분석 요청은 다음과 같습니다.

```text
당사 영향 중심으로 핵심 판단, 리스크 수준, 후속 확인사항을 정리해줘.
```

```text
영업현장에 영향을 줄 수 있는 내용을 중심으로 요약해줘.
```

```text
임원 보고용으로 5줄 요약과 확인 포인트를 정리해줘.
```

보도자료 분석은 반드시 근거 보도자료 번호와 함께 보는 것이 좋습니다.

### 12-10. 리스크 대응센터 사용 방법

리스크 대응센터는 부정 또는 주의 기사에 대한 대응 초안을 만드는 곳입니다.

사용 순서는 다음과 같습니다.

1. 왼쪽 메뉴에서 `리스크 대응센터`를 클릭합니다.
2. 부정 또는 주의 기사 목록을 확인합니다.
3. 대응이 필요한 기사를 선택합니다.
4. 기사 URL을 붙여넣거나 드래그합니다.
5. 사실관계나 내부 확인 내용을 입력합니다.
6. 원하는 용도를 선택합니다.

   예시는 다음과 같습니다.

   - 사내 공유용
   - 임원 보고용
   - 언론 문의 대응용
   - 영업현장 안내용

7. AI 초안 생성을 실행합니다.
8. 나온 초안을 그대로 쓰지 말고 사실관계를 확인한 뒤 수정합니다.

리스크 대응 초안은 “완성 문서”가 아니라 “초안”입니다.

특히 법적 표현, 당사 입장, 사실관계는 반드시 사람이 확인해야 합니다.

보고서는 일일, 주간, 월간으로 나뉩니다.

- 13:00: 당일 08:00부터 13:00까지
- 18:00: 당일 13:00부터 18:00까지

확인할 내용은 다음입니다.

- 당사 부정 여부
- 주의 기사
- 정책/규제 기사

- 분석 근거 기사

확인할 내용은 다음입니다.

- 월간 총 기사량
- 당사 기사
- GA/보험사 동향
- 정책/규제 흐름
- 부정/주의 기사
- 언론사 활동
- 핵심 키워드
- 다음 달 관리 포인트

### PDF 저장

보고서 화면에서 `인쇄 / PDF 저장` 버튼을 누릅니다.

브라우저 인쇄창이 뜨면 프린터를 `PDF로 저장`으로 선택합니다.

파일명은 아래처럼 저장하면 관리하기 쉽습니다.

```text
2026-06-01_일일_언론동향보고서.pdf
2026-06_월간_미디어모니터링보고서.pdf
2026-W22_주간_미디어모니터링보고서.pdf
```

### 12-12. 알림톡 발송 이력 확인 방법

알림톡 발송 이력은 실제로 카카오 알림이 나갔는지 확인하는 곳입니다.

1. 통합 대시보드에서 `알림톡 발송 이력`을 봅니다.
2. 최신 발송 이력 3개가 기본으로 보여야 합니다.
3. 더 많이 보고 싶으면 `발송 이력 더보기`를 누릅니다.
4. 특정 발송 이력을 누르면 당시 발송된 알림톡 본문을 확인합니다.
5. 보고서 링크가 있으면 해당 보고서를 열어봅니다.

발송 이력은 주간, 월간, 일일 순서로 고정되어 있으면 안 됩니다.

반드시 실제 발송 시간 기준 최신순이어야 합니다.

### 12-13. 언론사 관리 방법

언론사 관리는 단순 목록이 아니라 관계 관리용 화면입니다.

확인 순서는 다음입니다.

1. 왼쪽 메뉴에서 `언론사 관리`를 클릭합니다.
2. 보도량이 많은 언론사를 확인합니다.
3. 당사 언급 수를 확인합니다.
4. 부정 또는 주의 논조가 있었는지 확인합니다.
5. 관리가 필요한 언론사는 `관리` 버튼을 누릅니다.
6. 등급, 담당자, 메모, 최근 접촉일을 입력합니다.
7. 언론사 주소가 잘못 잡히면 주소 보정을 등록합니다.

예를 들어 `insnews.co.kr`이 실제로 보험매일이면, 언론사 관리에서 해당 주소를 보험매일로 보정합니다.

이후 같은 주소로 들어오는 기사는 보험매일로 표시되도록 관리합니다.

### 12-14. 기자 관리 방법

기자 관리는 필요한 기자만 열어서 관리합니다.

1. 왼쪽 메뉴에서 `기자 관리`를 클릭합니다.
2. 기자 목록을 확인합니다.
3. 수정할 기자의 `관리` 버튼을 누릅니다.
4. 소속, 관계 상태, 최근 접촉일, 메모를 수정합니다.
5. 저장합니다.

기자 관리 화면은 항상 길게 펼쳐두는 화면이 아니라, 필요한 항목만 열어 관리하는 방식입니다.

### 12-15. 광고비/협찬 관리 방법

광고비와 협찬 관리는 언론사 관계와 예산 집행을 함께 보기 위한 기능입니다.

1. 왼쪽 메뉴에서 `광고비/협찬 관리`를 클릭합니다.
2. 연도, 월, 언론사, 구분을 선택합니다.
3. 선택 기간의 총 집행액을 확인합니다.
4. 월별 집행 추이를 봅니다.
5. 매체별 집행 현황을 봅니다.
6. 필요하면 `집행 추가`를 눌러 새 내역을 입력합니다.
7. 보고가 필요하면 보고서 인쇄 기능을 사용합니다.

이 화면은 단순 입력 화면이 아니라 예산 현황판처럼 쓰는 것이 목표입니다.

### 12-16. 자동화가 제대로 도는지 확인하는 방법

자동화가 정상인지 확인할 때는 세 가지를 봅니다.

### 1. 대시보드 데이터

최근 기사와 주요 이슈가 갱신되고 있는지 확인합니다.

### 2. 알림톡 발송 이력

08:00, 13:00, 18:00 일일보고가 정상적으로 남는지 확인합니다.

주간 보고서는 월요일 07:00, 월간 보고서는 매월 1일 07:00에 남아야 합니다.

### 3. GitHub Actions

GitHub 저장소의 `Actions` 메뉴에서 실행 결과를 확인합니다.

주요 workflow는 다음입니다.

- `news-briefing.yml`: 일일, 주간, 월간 보고서
- `negative-watch.yml`: 5분 단위 부정기사 감시
- `regulator-releases.yml`: 금융당국 보도자료 수집
- `pages-dashboard.yml`: 대시보드 배포

실패가 있으면 실패한 workflow를 열고 로그를 확인합니다.

### 12-17. 문제가 생겼을 때 빠른 점검표

### 화면이 안 바뀌는 경우

1. 브라우저 새로고침
2. GitHub Actions 배포 완료 여부 확인
3. Supabase 연결 상태 확인
4. 최신 커밋이 main에 반영됐는지 확인

### 알림톡이 안 오는 경우

1. 알림톡 발송 이력 확인
2. GitHub Actions 실행 여부 확인
3. Kakao refresh token 만료 여부 확인
4. 해당 시간대에 발송 조건이 있었는지 확인

### 기사가 너무 이상하게 들어오는 경우

1. 키워드가 너무 넓은지 확인
2. 업종 문맥 단어를 붙임
3. 제외 후보 키워드 등록 검토
4. 같은 오류가 반복되면 분석 로직 수정 필요

### 요약이 이상한 경우

1. 원문 본문이 제대로 수집됐는지 확인
2. RSS 요약만 들어온 것은 아닌지 확인
3. 기자명, 사진 설명, 도입 문구가 섞였는지 확인
4. 필요하면 원문 본문 추출 로직 개선

### 언론사명이 주소로 나오는 경우

1. 언론사 관리 화면에서 해당 주소 확인
2. 관리 버튼 클릭
3. 실제 언론사명 입력
4. 주소 보정 저장

### 12-18. 상사에게 설명할 때 쓰는 요약

상사에게 이 프로젝트를 설명할 때는 기술 용어보다 업무 효과 중심으로 설명하는 것이 좋습니다.

예시는 다음과 같습니다.

```text
이 시스템은 마케팅부 언론홍보 업무에서 반복적으로 하던 뉴스 검색과 이슈 확인을 자동화한 도구입니다.
관심 키워드를 등록하면 관련 뉴스와 금융당국 보도자료를 자동 수집하고, AI가 당사 언급, 부정/주의 이슈, 정책/규제 동향을 분류합니다.
결과는 실시간 대시보드, 일일/주간/월간 보고서, 카카오 알림으로 확인할 수 있습니다.
이를 통해 담당자가 매번 직접 검색하지 않아도 주요 언론 동향과 리스크 신호를 빠르게 확인할 수 있고, 누적 데이터는 언론사 관리와 월간 분석에도 활용할 수 있습니다.
```

### 12-19. 인수인계 체크리스트

다른 사람이 이 업무를 이어받을 때는 아래 순서로 확인하면 됩니다.

1. 대시보드 주소를 열 수 있는가?
2. 환경 설정에서 키워드를 볼 수 있는가?
3. 실시간 모니터링에서 기사 목록이 보이는가?
4. 스크랩 기능이 작동하는가?
5. 금융당국 보도자료가 보이는가?
6. 리스크 대응센터에서 초안 생성이 되는가?
7. 알림톡 발송 이력이 최신순으로 보이는가?
8. 일일, 주간, 월간 보고서 링크가 열리는가?
9. GitHub Actions 실행 내역을 확인할 수 있는가?
10. Supabase에서 데이터가 누적되는지 확인할 수 있는가?
11. Kakao, Gemini, Supabase, GitHub 관련 비밀값이 어디에 저장되어 있는지 알고 있는가?
12. 문제가 생겼을 때 누구에게 공유해야 하는지 알고 있는가?

이 체크리스트를 통과하면 기본 운영은 할 수 있습니다.

## 13. 개발자가 이어받을 때

처음 개발 환경을 준비할 때는 저장소를 내려받습니다.

```powershell
git clone https://github.com/incarmarketing/news-monitor.git
cd news-monitor
```

이미 내려받은 폴더가 있다면 최신 내용을 가져옵니다.

```powershell
git pull
```

프론트엔드 화면을 수정했다면 빌드합니다.

```powershell
npm.cmd --prefix frontend run build
```

자동화와 배포는 GitHub Actions를 통해 실행됩니다.

중요한 비밀값은 코드에 쓰지 말고 GitHub Secrets 또는 Supabase Secrets에 넣어야 합니다.

필요한 비밀값 이름은 보통 다음과 같습니다.

- GEMINI_API_KEY
- KAKAO_REST_API_KEY
- KAKAO_REFRESH_TOKEN
- SUPABASE_URL
- SUPABASE_SERVICE_ROLE_KEY
- SUPABASE_ANON_KEY
- CRONJOB_API_KEY

값 자체는 문서에 적지 않습니다.

## 14. 이 프로젝트의 목표

이 프로젝트의 목표는 화려한 화면 하나를 만드는 것이 아닙니다.

반복 검색을 줄이고, 놓치기 쉬운 부정 또는 주의 신호를 빨리 확인하고, 자료를 계속 쌓아 일일/주간/월간 보고서로 활용할 수 있게 만드는 것입니다.

즉, 목표는 이것입니다.

```text
매번 검색하지 않아도 된다.
중요한 이슈를 놓치지 않는다.
보고서 작성 시간이 줄어든다.
데이터가 계속 쌓인다.
다른 업무에도 확장할 수 있다.
```

이 문서만 보고도 큰 흐름을 이해했다면 충분합니다.

세부 기능은 대시보드를 직접 눌러보면서 익히면 됩니다.
