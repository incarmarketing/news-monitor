# 커스터마이즈 상세 가이드

이 문서는 템플릿을 자기 업무에 맞게 바꾸는 방법을 설명합니다. 핵심은 “수집 대상”, “분류 기준”, “보고서 표현”을 분리해서 생각하는 것입니다.

## 1. 먼저 업무 질문을 정합니다

처음부터 키워드만 넣으면 노이즈가 많이 생깁니다. 아래 질문에 먼저 답합니다.

| 질문 | 예시 |
| --- | --- |
| 매일 확인하기 귀찮은 자료는 무엇인가? | 뉴스, 보도자료, 공시, 리뷰, 경쟁사 공지 |
| 누가 이 결과를 보는가? | 실무자, 팀장, 임원, 영업 현장 |
| 무엇이 나오면 바로 알아야 하는가? | 부정 기사, 제재, 고객 불만, 경쟁사 신제품 |
| 무엇은 제외해야 하는가? | 동명이인, 스포츠, 채용, 단순 포토, 무관 산업 |
| 결과는 어떤 형태가 좋은가? | 카톡 요약, 대시보드, 일일 보고서, 월간 리포트 |

## 2. 키워드 구조를 설계합니다

키워드는 하나의 큰 목록이 아니라 역할별로 나눠야 합니다.

| 구분 | 목적 | 좋은 예 |
| --- | --- | --- |
| 당사 | 직접 언급 확인 | 회사명, 브랜드명, 제품명 |
| 경쟁사 | 비교 대상 추적 | 경쟁사명, 대체 서비스명 |
| 업계 | 시장 흐름 파악 | 업종명, 상품군, 채널명 |
| 정책/규제 | 제도 변화 감지 | 감독기관, 법안명, 제도명 |
| 제외 | 노이즈 제거 | 무관 스포츠, 채용, 동명이인 |

짧고 흔한 단어는 단독 키워드로 쓰지 않는 것이 좋습니다. 예를 들어 `메가`처럼 여러 맥락에서 쓰이는 단어는 `메가 + 업종 문맥` 조건이 붙어야 합니다.

## 3. 설정 파일을 바꿉니다

가장 쉬운 방법은 `.env`에서 바꾸는 것입니다.

```env
COMPANY_NAME=내회사명
TEAM_NAME=내부서명
OWN_NAMES=내회사명,내서비스명
MONITOR_KEYWORDS=내회사명,내서비스명,경쟁사명,업계 키워드,정책 키워드
EXCLUDE_KEYWORDS=채용,구인,무관 스포츠,무관 이벤트
```

더 정교하게 관리하려면 Supabase의 `monitor_keywords` 테이블과 대시보드의 환경 설정 화면에서 키워드를 관리합니다.

## 4. 분석 기준을 바꿉니다

`analyzer.py`에서 아래 목록을 업무에 맞게 바꿉니다.

| 항목 | 역할 |
| --- | --- |
| `REGULATION_WORDS` | 정책/규제 신호 |
| `COMPETITOR_WORDS` | 경쟁사/시장 신호 |
| `DOMAIN_CONTEXT_WORDS` | 업종 관련성 판단 |
| `NEGATIVE_WORDS` | 일반 부정 신호 |
| `SEVERE_NEGATIVE_WORDS` | 즉시 대응이 필요한 강한 부정 신호 |
| `POSITIVE_WORDS` | 성과나 우호 보도 신호 |

분류가 자꾸 틀리면 키워드만 추가하지 말고, “왜 무관한 자료가 들어왔는지”를 먼저 봅니다. 보통은 업종 문맥 필터가 약하거나 제외 키워드가 부족한 경우가 많습니다.

## 5. 수집 대상을 바꿉니다

현재 기본 수집기는 뉴스 중심입니다.

| 파일 | 역할 |
| --- | --- |
| `news_collector.py` | 네이버/구글 뉴스 수집 |
| `regulator_collector.py` | 기관 보도자료 수집 |
| `refresh_regulator_releases.py` | 보도자료 갱신 |

다른 자료를 모으려면 새 수집기를 하나 추가하고, 결과를 기존 기사 형식에 맞춥니다.

```json
{
  "title": "자료 제목",
  "link": "원문 링크",
  "source": "출처",
  "keyword": "검색 키워드",
  "description": "본문 또는 요약",
  "pub_date": "발행일"
}
```

이 형식으로만 맞추면 뒤의 분석, 저장, 보고서 흐름은 재사용하기 쉽습니다.

## 6. 보고서 문장을 바꿉니다

| 파일 | 역할 |
| --- | --- |
| `ai_briefing.py` | 일일 브리핑 문장 |
| `period_report.py` | 주간/월간 보고서 문장 |
| `dashboard_builder.py` | HTML 보고서 구성 |
| `templates/period_report.html` | 기간 보고서 레이아웃 |
| `templates/dashboard.html` | 대시보드 레이아웃 |

보고서 문장은 “멋진 말”보다 “근거와 판단이 분리된 문장”이 중요합니다.

좋은 보고서 문장의 기준:

- 핵심 이슈가 실제 기사 목록에 반드시 존재한다.
- 기사 제목만 반복하지 않고 본문 맥락을 요약한다.
- 긍정/중립/주의/부정 기준이 일관된다.
- 당사 직접 이슈와 업계 일반 이슈를 섞지 않는다.
- 임원이 봐도 “그래서 무엇을 봐야 하는지”가 보인다.

## 7. 자동 실행을 연결합니다

기본 자동화는 GitHub Actions입니다.

| 워크플로우 | 역할 |
| --- | --- |
| `news-briefing.yml` | 일일/주간/월간 보고서 생성 |
| `negative-watch.yml` | 부정/주의 이슈 감시 |
| `pages-dashboard.yml` | 대시보드 배포 |
| `sync-cronjob.yml` | cron-job.org 작업 동기화 |

PC가 꺼져도 돌아가게 하려면 GitHub Actions와 Supabase, cron-job.org를 연결합니다. 로컬 PC는 개발과 테스트용으로만 둡니다.

## 8. 공유용 저장소로 만들 때

공유용 저장소에는 운영 데이터가 없어야 합니다.

남기면 안 되는 것:

- `.env`
- `logs/`
- `data/`
- `public/`
- `.run-state/`
- `.watch-state/`
- 실제 보고서 HTML
- 실제 발송 이력 JSON
- 실제 고객/회사/사람 정보

공유용 저장소에 남겨도 되는 것:

- 코드
- 템플릿 HTML
- 예시 프로필
- 예시 환경 파일
- SQL 스키마
- 워크플로우 파일
- 사용 설명서

## 9. 좋은 확장 방향

| 확장 | 설명 |
| --- | --- |
| 리뷰 분석 | 고객 리뷰를 모아 불만 유형과 개선 포인트를 자동 분류 |
| 경쟁사 레이더 | 경쟁사 보도자료, 채용, 블로그를 모아 월간 동향 작성 |
| 정책 리스크 | 기관 보도자료와 법안을 모아 영업/상품 영향도 분석 |
| 영업 현장 VOC | 내부 문의나 민원 로그를 모아 반복 이슈를 대시보드화 |
| 광고/협찬 관리 | 매체별 집행액, 보도량, 우호도, 접촉 이력을 함께 관리 |

핵심은 같은 구조를 재사용하는 것입니다. 수집기만 바꾸고, 데이터 형식과 분석·보고서 흐름은 최대한 유지합니다.
