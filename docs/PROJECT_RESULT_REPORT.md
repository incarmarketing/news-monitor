# AI 뉴스 모니터링 자동화 프로젝트 결과보고서

작성일: 2026-06-07  
대상 프로젝트: `news-monitor`  
운영 기준 브랜치: `main`  
운영 URL: https://incarmarketing.github.io/news-monitor/dashboard.html

## 1. Executive Summary

`news-monitor`는 인카금융서비스 홍보/마케팅 업무에서 반복적으로 발생하는 언론 모니터링, 기사 분류, 부정기사 감시, 일일/주간/월간 보고, 카카오톡 알림, 리스크 대응 초안 작성을 하나의 자동화 흐름으로 연결하기 위해 구축한 프로젝트입니다.

초기 목표는 "뉴스를 자동으로 모으고 보고서를 보내는 도구"였지만, 현재 구조는 단순 뉴스 수집기를 넘어 `인카 PR Control Center`에 가까운 형태로 발전했습니다. 핵심은 사람이 직접 검색하고, 분류하고, 보고서를 만들고, 부정 이슈를 놓치지 않기 위해 확인하던 과정을 클라우드 기반 자동화로 대체하는 것입니다.

현재 프로젝트는 다음 기능을 중심으로 운영됩니다.

- 키워드 기반 기사 자동 수집
- 당사/경쟁사/GA/보험사/정책/규제 등 문맥 기반 분류
- 긍정/중립/주의/부정 논조 판단
- 같은 이슈 기사 묶음 처리
- 실시간 대시보드와 미디어 분석 리포트
- 일일/주간/월간 보고서 생성 및 GitHub Pages 배포
- 카카오톡 보고서 발송
- 24시간 부정기사 감시
- 알림톡 발송 이력과 운영 헬스체크
- Gemini/Groq/규칙 기반 요약 백업 구조
- 언론사/기자/광고비/스크랩/리스크 대응센터 기반 확장 구조

이 프로젝트의 가장 중요한 성과는 로컬 PC 한 대에서만 동작하는 자동화가 아니라, GitHub Actions, cron-job.org, Supabase, GitHub Pages를 연결해 PC가 꺼져 있어도 운영이 이어지는 구조를 만든 점입니다.

## 2. 기획 의도

홍보 업무의 어려움은 단순히 기사를 많이 보는 데 있지 않습니다. 중요한 것은 수많은 기사 중에서 회사와 직접 관련된 이슈, 업계 흐름, 정책 변화, 부정 리스크, 대응 필요 기사를 빠르게 구분하는 것입니다.

기존 수작업 방식에서는 다음 문제가 반복됩니다.

- 검색 키워드가 많아질수록 누락 가능성이 커집니다.
- 같은 기사가 여러 언론사에서 반복 노출되어 실제 이슈 규모를 파악하기 어렵습니다.
- `글로벌금융`, `메가`, `AI`, `브랜드평판`처럼 짧거나 범용적인 키워드는 무관 기사를 많이 끌고 옵니다.
- 부정기사와 단순 주의 기사가 섞이면 리스크 레벨을 신뢰하기 어렵습니다.
- 보고서 작성 시간이 오래 걸리고, 일일/주간/월간 기준이 흔들릴 수 있습니다.
- 알림 발송 여부와 보고서 링크가 정확히 남지 않으면 운영 신뢰도가 떨어집니다.
- 특정 PC나 담당자 환경에 의존하면 출근 장소가 바뀔 때 업무가 끊깁니다.

따라서 이 프로젝트의 기획 의도는 명확합니다.

1. 반복 검색을 자동화합니다.
2. 무관 기사와 중복 기사를 줄입니다.
3. 회사 기준에 맞는 문맥 판단을 쌓습니다.
4. 대시보드와 보고서를 통해 의사결정에 필요한 정보만 보여줍니다.
5. 부정기사 감시와 보고서 발송은 클라우드에서 안정적으로 실행합니다.
6. 모든 데이터는 로컬이 아닌 Supabase 중심으로 누적합니다.
7. 향후 언론 CRM, 보도자료 배포, 광고비 관리, AI 업무 지원까지 확장 가능한 구조로 만듭니다.

## 3. 프로젝트 빌드업 과정

### 3.1 1단계: 수집 자동화

초기에는 네이버/구글 뉴스 기반으로 키워드 검색 결과를 수집하는 구조를 만들었습니다. 기사 제목, 링크, 출처, 발행일, 키워드, 요약 원문을 표준 형태로 정리하고, 같은 링크나 거의 같은 제목은 중복으로 처리하도록 했습니다.

이 단계에서 중요한 개선은 단순 키워드 검색의 한계를 인식한 것입니다. 예를 들어 `메가`, `글로벌금융`, `브랜드평판`처럼 의미가 넓은 키워드는 보험/GA/금융 문맥과 함께 등장할 때만 의미 있는 기사로 보도록 기준을 강화했습니다.

### 3.2 2단계: 문맥 기반 분석

수집 기사에 대해 당사, 경쟁사, GA, 보험사, 정책/규제, 업계 동향 등으로 분류하는 분석 로직을 만들었습니다. 이후 긍정/중립/주의/부정 논조를 분리했고, 직접 부정 기사와 시장성·정책성 주의 이슈를 구분하도록 개선했습니다.

특히 부정기사 기준은 계속 다듬었습니다. 투자의견 하향, 목표가 조정, 손해율 악화, 수수료 규제, 정착지원금 공시처럼 직접 부정은 아니지만 관찰이 필요한 기사는 `주의`로 분리했습니다. 반대로 사칭, 사고, 제재, 내부통제, 법적 분쟁처럼 즉시 대응이 필요한 경우만 `부정`에 가깝게 보도록 방향을 잡았습니다.

### 3.3 3단계: 대시보드 구축

초기 대시보드는 수집 결과를 보여주는 화면에 가까웠지만, 이후 실시간 상황판으로 개편했습니다. 현재 대시보드는 단순 기사 목록이 아니라 오늘의 상태, 주요 이슈, 부정기사 감시, 알림 발송 이력, 보고서 자동화, API 사용 현황, 미디어 분석 리포트를 한 곳에서 확인하는 운영 화면을 목표로 합니다.

디자인 측면에서는 모바일 화면 넘침, 버튼/라벨 겹침, 그래프 가독성, 카드 간격, 보고서 인쇄/PDF 품질, 주요 이슈 요약 품질을 반복적으로 개선했습니다. 기능이 많아질수록 화면이 복잡해지는 문제가 있어, 앞으로도 "첫 화면은 판단에 필요한 것만 남긴다"는 원칙을 유지해야 합니다.

### 3.4 4단계: 보고서 자동화

일일 보고서는 08시, 13시, 18시 기준으로 나누고, 주간 보고서는 매주 월요일 07시, 월간 보고서는 매월 1일 07시에 발송하는 구조로 정리했습니다. 보고서 링크는 GitHub Pages에 배포되며, 카카오톡 메시지에서 바로 열 수 있도록 했습니다.

보고서 품질 개선 과정에서 가장 큰 방향 전환은 "대시보드 화면을 PDF로 찍는 방식"이 아니라, 일간지/주간지/월간지처럼 읽히는 별도 산출물로 만드는 것이었습니다. 따라서 보고서는 핵심 수치, 대표 기사, 기간별 논조 추이, 언론사 영향도, 키워드별 기사량, 핵심 이슈 요약을 중심으로 구성했습니다.

### 3.5 5단계: 부정기사 24시간 감시

부정기사 감시는 처음에는 5분 범위 검사와 실행 주기 사이에 혼선이 있었습니다. 이후 기본 원칙을 `24시간 감시`, `5분 단위 실행`, `최근 성공 시각 기준 보정 검사`로 정리했습니다.

현재 구조는 GitHub Actions가 감시를 수행하고, cron-job.org가 외부에서 워크플로우를 깨우며, Supabase Cron이 보조 백업으로 감시 상태를 확인하는 구조입니다. 대시보드에는 마지막 수행 시각, 상태, 신규 기사 수, 성공/실패 여부가 기록됩니다.

### 3.6 6단계: 클라우드 운영 안정화

이 프로젝트에서 중요한 전환은 로컬 PC 의존을 줄인 것입니다. GitHub Actions, Supabase, GitHub Pages, cron-job.org를 연결하면서 집 PC, 회사 PC, 로컬 실행 여부와 무관하게 운영할 수 있는 기반을 만들었습니다.

현재 운영 구조는 다음과 같습니다.

```text
키워드/설정
  Supabase
      ↓
수집/분석 실행
  GitHub Actions + cron-job.org + Supabase Cron
      ↓
데이터 누적
  Supabase tables
      ↓
화면/보고서 생성
  GitHub Pages
      ↓
알림
  KakaoTalk link message
```

## 5. 대시보드 설계 의도

대시보드는 많은 기능을 넣는 화면이 아니라, 홍보 담당자가 지금 무엇을 봐야 하는지 알려주는 상황판이어야 합니다. 따라서 화면 설계의 핵심 기준은 다음과 같습니다.

- 첫 화면에서는 오늘 상태를 빠르게 판단할 수 있어야 합니다.
- 부정기사 감시와 보고서 자동화 상태는 항상 신뢰 가능해야 합니다.
- 주요 이슈는 기사 본문을 그대로 잘라 붙이지 않고 핵심만 요약해야 합니다.
- 같은 이슈를 여러 기사로 반복 표기하지 않아야 합니다.
- 모바일에서도 버튼, 라벨, 날짜 필터가 넘치지 않아야 합니다.
- 보고서는 대시보드와 별도 산출물로 보여야 합니다.
- 숫자는 많아도 문장은 짧아야 합니다.

현재 대시보드는 `실시간 상황판`, `모니터링`, `미디어 분석`, `리스크 대응`, `운영 관리`를 통합하는 방향으로 설계되어 있습니다. 향후에는 사용자 권한과 역할에 따라 첫 화면에서 보이는 카드와 알림 기준을 다르게 구성할 수 있습니다.

## 6. 확장성

이 프로젝트는 뉴스 모니터링만을 위한 고정형 코드가 아니라, 반복 검색과 판단을 자동화하는 범용 구조로 확장할 수 있습니다.

### 6.1 다른 데이터 소스로 확장

현재는 뉴스 중심이지만 아래 데이터 소스로 확장 가능합니다.

- 금융위원회/금융감독원 보도자료와 첨부 PDF
- 보험협회/GA협회 공시 자료
- 경쟁사 홈페이지 공지
- 공시/IR 자료
- 블로그/SNS/커뮤니티
- 고객 리뷰/민원 데이터
- 내부 상담 로그 또는 사내 보고 문서

핵심은 `수집기`, `분류 기준`, `보고서 문구`를 해당 데이터 소스에 맞게 교체하는 것입니다.

### 6.2 다른 부서 업무로 확장

홍보팀 외에도 다음 업무에 활용할 수 있습니다.

- 영업기획: 경쟁사 상품/수수료/조직 동향
- 준법감시: 감독기관 공지와 제도 변화
- 소비자보호: 민원/분쟁/불완전판매 이슈
- 경영기획: 공시/IR/시장평가 변화
- 마케팅: 브랜드 언급, 캠페인 노출, 언론 반응
- 인사/조직: 채용공고, 조직개편, 업계 인재 이동

### 6.3 알림 수신자 확장

현재 카카오톡 알림은 개인 수신 중심으로 운영됩니다. 향후에는 다음 구조로 확장하는 것이 적절합니다.

- 수신자 그룹 관리
  - 홍보팀
  - 마케팅팀
  - 임원
  - 리스크 대응 담당자
  - 주간/월간 보고 수신자
- 알림 유형별 수신자 분리
  - 일일 언론 동향
  - 주간/월간 보고서
  - 부정기사 즉시 알림
  - 정책/규제 중요 알림
  - 시스템 오류 알림
- 수신자별 발송 조건
  - 평일만 수신
  - 업무시간만 수신
  - 부정기사만 수신
  - 당사 언급 기사만 수신
- 발송 이력 상세화
  - 수신자별 성공/실패
  - 재전송 여부
  - 링크 클릭 여부
  - 열람 시각
  - 메시지 원문

이 기능이 들어가면 알림톡은 단순 개인 알림이 아니라 PR 운영의 공식 커뮤니케이션 로그가 됩니다.

### 6.4 언론 CRM으로 확장

언론사 관리와 기자 관리는 현재 목록 관리 단계에서 관계 관리 단계로 확장할 수 있습니다.

- 당사 보도자료를 한 번이라도 낸 언론사 자동 등록
- 언론사별 당사 기사, 부정 기사, 광고비, 접촉 이력 연결
- 기자별 담당 분야, 연락처, 메모, 선호 이슈 관리
- 보도자료 발송 대상 추천
- 후속 보도 매칭
- 언론사 영향도 점수와 관계 등급 관리

이 방향으로 발전하면 프로젝트는 뉴스 모니터링 도구를 넘어 PR CRM이 됩니다.

### 6.5 AI Workbench로 확장

현재 AI는 기사 요약과 대응 초안에 주로 사용됩니다. 향후에는 다음 기능을 묶어 AI Workbench로 확장할 수 있습니다.

- 보도자료 초안
- 해명자료 초안
- 사내 보고용 요약
- 임원 보고 문구
- 기자 피칭 메일
- FAQ/Q&A 생성
- 기사 반박 포인트 정리
- 기존 작성물 문체 학습
- 생성 결과 버전 저장과 비교

중요한 점은 AI가 판단을 대신하는 것이 아니라, 사람이 더 빨리 판단하도록 초안과 근거를 정리하는 역할을 해야 한다는 것입니다.

## 7. 운영 안정성

운영 안정성은 이 프로젝트의 핵심입니다. 보고서 발송과 부정기사 감시는 오류가 나면 신뢰를 잃기 때문에, 다음 구조로 안정성을 높였습니다.

### 7.1 자동 실행 구조

- GitHub Actions: 기본 실행자
- cron-job.org: 예약 실행 지연 보완
- Supabase Cron: 외부 cron까지 지연될 때 보조 백업
- schedule_guard: 중복 발송 방지와 누락 감지
- job_ledger: 실행 이력 기록
- notification history: 카카오 발송 성공/실패 기록

### 7.2 데이터 저장 구조

운영 데이터는 로컬 브라우저 저장이 아니라 Supabase에 누적하는 방향입니다.

- 기사 데이터
- 분석 결과
- 키워드 설정
- 스크랩
- 언론사/기자 정보
- 광고비 관리
- 보고서 실행 이력
- 알림톡 발송 이력
- 부정기사 감시 이력

### 7.3 API 장애 대응

AI API 사용량 초과나 오류가 발생할 수 있기 때문에 다음 방향을 적용했습니다.

- Gemini 사용량 문제 발생 시 Groq 백업 사용
- Groq도 실패하면 규칙 기반 요약 사용
- 무한 재시도 방지
- 실패 상태를 운영 화면에 표시
- API 사용 현황 카드로 조기 인지

## 8. 남은 과제와 개선 방향

### 8.1 분석 품질 고도화

- 당사 직접 영향과 업계 일반 이슈를 더 정밀하게 구분
- `주의`와 `부정`의 기준을 더 명확히 분리
- 짧은 키워드의 문맥 필터 강화
- 기사 요약의 문장 품질 고도화
- 같은 이슈 묶음 대표 기사 선정 기준 개선
- 금융당국 보도자료 첨부 PDF 분석
- AI와 규칙 기반 분석 결과의 품질 차이 최소화

### 8.2 보고서 품질 고도화

- 일일/주간/월간 보고서의 역할 명확화
- 주간/월간은 기간 인사이트 중심으로 강화
- 보고서 내 기사 요약 중복 제거
- 보고서 링크가 항상 최신 슬롯과 연결되도록 검증
- 인쇄/PDF 저장 시 페이지 넘김과 여백 안정화
- 보고서 생성 실패 시 자동 재시도와 오류 알림

### 8.3 대시보드 UX 고도화

- 모바일 필터 영역 넘침 완전 차단
- 주요 이슈 카드의 요약 품질 개선
- 그래프 유형과 간격 재정리
- 작업 중 상태를 화면 전환과 무관하게 유지
- 더보기 버튼과 팝업 기능 완성도 강화
- 기사 열기는 항상 새 탭으로 열기
- 언론사 영향도 클릭 시 기간 조건 유지

### 8.4 운영 관리 고도화

- 알림 수신자 다중 등록
- 수신자 그룹/권한/알림 유형 관리
- 알림 실패 시 관리자 알림
- 카카오 메시지 원문 팝업 확인
- API 사용량 임계치 알림
- 보고서 생성/발송 누락 자동 복구
- GitHub Actions, cron-job.org, Supabase Cron 상태 통합 표시

### 8.5 언론 CRM과 배포 성과

- 언론사 상세 관리 화면 강화
- 기자별 정보 입력과 접촉 이력 관리
- 당사 보도 언론사 자동 DB 등록
- 보도자료 배포 캠페인 생성
- 발송 성공/클릭/후속 보도 매칭
- 광고비와 보도 성과 연결
- 언론사 영향도 점수와 관계 등급 도입

### 8.6 보안과 권한

- 공개 GitHub Pages 보고서의 접근 통제 검토
- 사용자 권한별 메뉴 노출
- API Key와 토큰 관리 기준 문서화
- Supabase RLS 정책 점검
- 로그에 민감 정보가 남지 않도록 점검
- 발송 대상과 보고서 열람 로그 관리

## 10. 결론

이 프로젝트는 단순한 자동 뉴스 수집기가 아니라, 홍보팀의 반복 업무를 줄이고 리스크 감지와 보고 품질을 높이기 위한 운영 플랫폼입니다. 현재까지의 작업을 통해 수집, 분석, 저장, 대시보드, 보고서, 알림, 감시, QA, 배포까지 하나의 흐름으로 연결했습니다.

앞으로의 핵심은 기능을 더 많이 붙이는 것이 아니라, 운영자가 믿고 쓸 수 있는 안정성과 분석 품질을 높이는 것입니다. 특히 보고서 발송 누락 방지, 부정기사 감시 신뢰도, AI 요약 품질, 알림 수신자 확장, 언론 CRM 고도화가 다음 성장의 중심이 됩니다.

최종 목표는 인카금융서비스 내부 기준에 최적화된 PR 운영센터입니다. 범용 PR 플랫폼을 그대로 따라가는 것이 아니라, 보험/GA/금융당국/당사 리스크 문맥을 깊게 반영한 사내 전용 도구로 발전시키는 것이 이 프로젝트의 가장 큰 방향입니다.
